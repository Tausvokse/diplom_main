# Інструкції для AI (Antigravity / Nexus)

Ці правила мають пріоритет під час виконання завдань:

## 1. Автоматизація Git
- Самостійно виконувати весь цикл Git-команд:
  - `git add` (додавання змінених файлів).
  - `git commit` (створення коммітів із логічними повідомленнями).
  - `git push` (відправка змін у репозиторій на GitHub).
- Виправляти помилки, що виникають під час деплою (наприклад, помилки TypeScript на Heroku), та відразу пушити виправлення.

## 2. Специфічне іменування файлів
- Жорсткий стандарт для зберігання документів у Supabase.
- Формат: `(НОМЕР СТУДЕНТСЬКОГО)_(Прізвище)_(тип_документа).extension`.
  - Приклад: `KB-123456_Ivanov_passport.pdf`.
- Прізвище має бути очищене від пробілів.

## 3. Логіка завантаження файлів (UX/UI)
- Змінена стандартна поведінка Multer та фронтенду.
- Обмеження: Заборона завантажувати декілька файлів в одне поле. Тільки один файл на категорію.
- Заміна: Якщо користувач обирає новий файл, він має автоматично замінювати попередній.
- Передогляд: Обов'язкова наявність візуального прев'ю (картинка для фото, іконка для PDF) перед завантаженням.

## 4. Робота з інфраструктурою (Supabase)
- Відмова від локальних папок: Повна заміна папки `uploads/` на хмарний Bucket `dormitory-files`.
- Безпека: Використання `service_role` ключа на бекенді для обходу RLS та коректна робота з Public URLs.
- Очищення конфігурації: Стежити, щоб у назвах бакетів чи посиланнях не було зайвих лапок (як це часто буває в `.env`).

## 5. Послідовність дій
- Спочатку детально розповідати теорію та кроки (наприклад, що натиснути в консолі Supabase).
- Тільки після цього переходити до написання коду.

## 6. Дані та БД
- Синхронізація ID: Використання фіксованих ID для категорій пільг (`id_orphans`, `id_disabled`, `id_combat`), щоб вони збігалися з кодом на фронтенді, а не були випадковими UUID.
